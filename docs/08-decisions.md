# 08 · 决策记录

> Typex 产品设计书 · 第八章
> 本章只记录已定决策（ADR），用于说明当前规格背后的关键取舍。

---

## 已定决策（ADR）

### ADR-1 · F-3 不做完整 AI Agent，定为单轮问答（2026-07-03）

- **背景**：最初设想接入 OpenAI Codex SDK 作为内嵌 Agent 引擎。调研发现：Codex SDK 是 codex CLI（15–25 MB/平台）的子进程包装；2026-02 起仅支持 Responses wire 协议，接第三方 OpenAI 兼容端点需网关中转；配置只认用户级 `~/.codex/config.toml`；其核心价值（OS 沙箱文件/命令操作、ChatGPT 订阅额度）与 Typex 的文本场景错配。同时市面已有 Codex、Claude Code 等成熟 Agent 产品，重造/内嵌轮子价值低。
- **决策**：F-3 = 单次 LLM 调用（选中文本处理 + 单轮问答），不引入 Agent SDK、不做工具循环与多轮会话。
- **复审条件**：若未来用户强烈要求「让助手操作本地文件/执行任务」，重新评估外部 Agent 接入（届时优先考虑 app-server / Agent SDK 的子进程集成，`providers/` 结构已预留扩展位）。

### ADR-2 · 语音输入流水线必须包含轻量文本整理（去噪）层（2026-07-03）

- **背景**：口语转写原样上屏不可用（语气词/重复/改口）；竞品调研确认「重写而非转写」是品类核心价值。
- **决策**：F-9 文本整理是 F-1 主干能力（STT → 整理 → 注入），使用独立的「文本整理」模型槽（推荐轻量快模型）；默认开启、一键可切原样模式、失败自动降级原文。

### ADR-3 · 桌面框架采用 Tauri 2 + Rust（2026-07-03）

- **背景**：低占用 + 三平台 + 系统级能力（全局键、注入、AX）需求；Electron 内存不达标，Flutter 桌面集成弱，原生三写维护不可行；同类开源项目（Handy/Whispering）已验证路线。
- **决策**：Tauri 2.x，系统能力全部在 Rust 侧实现（关键 crate 清单见 [06 §1](06-code-architecture.md)）。

### ADR-4 · 模型接入以 OpenAI 事实标准为默认协议 + 专用 adapter（2026-07-03）

- **决策**：STT 默认 `openai_compat`（multipart /audio/transcriptions），LLM 支持 `chat_completions` 与 `responses` 两种 wire；火山/豆包走专用 adapter（极速版 flash，双凭据）；流式转写延后（各家协议互不兼容，一次性转写 + 快 Provider 已够快）。依据见 [03 模型接入层](03-model-providers.md)。

### ADR-5 · BYOK、无账号、无中间服务器（2026-07-03）

- **决策**：Typex 不建服务端，用户自配端点与密钥；密钥与其他配置项一起本机存储，诊断导出与日志必须脱敏。除用户配置的端点与更新检查外零网络请求。这是产品的信任基石，写入 README（onboarding 首屏的承诺文案已于 2026-07-05 移除——界面保持克制，承诺由 README 与官网承载）。

### ADR-6 · 前端框架 Vue 3（2026-07-03，项目所有者拍板）

- **决策**：Vue 3 + TypeScript + Tailwind CSS v4 + Pinia（组合式 API + `<script setup>`）。理由：项目所有者熟悉度优先；Tauri 对框架无偏好，性能差异对本应用不构成瓶颈。同类项目（Svelte/React）仅借鉴架构思路。

### ADR-7 · 默认快捷键：全修饰键三角方案（2026-07-03 拍板，同日修订）

- **决策**：听写 = 按住 `右 ⌘`(macOS) / `右 Ctrl`(Win/Linux)；助手 = 按住 `右 ⌥` / `右 Alt`；翻译 = 按住 `右 ⌘ + 右 ⌥` / `右 Ctrl + 右 Alt`（两个触发键同按）。触发键按下乐观启动录音，组合出第二键无缝切为翻译。
- **理由**：不存在三平台通用的「无主之键」（Fn 被系统听写/中英切换占用且 Windows 硬件层不可见；CapsLock 是 macOS 中文用户的中英切换键；ScrollLock/Menu 键 Mac 键盘没有；带 Space/字母的组合键极易撞系统快捷键——首版方案 `右⌘+Space` 撞 Spotlight、`右Ctrl+Space` 撞输入法）。修饰键「单独按住」在几乎所有软件里是空操作，是冲突概率最低的键类。
- **修订记录**：首版为「助手 = 主键+Space」，因 Spotlight/输入法冲突需重度事件拦截，经项目所有者确认改为纯修饰键三角方案，事件拦截需求缩减为仅「Windows Alt 单击防菜单栏聚焦」一处。
- **附带义务**：① 组合键让路规则（触发键按住期间出现普通键 → 静默取消并放行，保证 `⌘C`/`AltGr+E` 等零干扰）；② Windows Alt 短按吞 keyup；③ 残余冲突（Karabiner/PowerToys 重映射、小配列键盘）靠 onboarding 改键 + HotkeyRecorder 冲突警告兜底。详见 [05 §7.1](05-ux-spec.md)、[06 §7.3](06-code-architecture.md)。

### ADR-8 · 历史记录默认开启（2026-07-03，项目所有者拍板）

### ADR-14 · 产品名写法统一为「Typex」（2026-07-03，项目所有者拍板）

- **决策**：一律写作 **Typex**（仅首字母大写，x 小写）；不用 TypeX / TYPEX / typeX。代码标识符按语言惯例（crate/包名 `typex`，类型名 `Typex...`）。logo、README、官网、文案全部遵循。

### ADR-15 · 仓库先私有开发，完善后公开（2026-07-03，项目所有者拍板）

- **决策**：开发期间仓库保持私有；待产品基本完善（至少达到「听写可用」质量，由项目所有者判断）再公开。公开即首发：配 README（GIF 演示）+ 发布帖，避免半成品消耗第一印象。
- **注意事项**：私有期间照常遵循 GPLv3 规范（LICENSE、依赖审计、Conventional Commits），公开时无需清理历史；第三方调研笔记等不宜公开的内容不入库。

### ADR-16 · 商业模式：软件开源免费 + 自营 STT 服务（2026-07-03，项目所有者拍板）

- **决策**：Typex 软件本体开源（GPLv3）、免费、**不内置任何收费功能**；盈利点为项目所有者后续自营的 **STT 套餐服务**——它以「官方 Provider 预设」的身份出现在模型服务列表中，与第三方预设（Groq、硅基流动、豆包……）**同列、可插拔、无特权**：不锁定、不强推、BYOK 永远是一等公民。
- **产品呈现**：官方服务在预设列表中排第一位并标注「官方 · 开箱即用」，价值主张是「不想研究 API 的用户，一个套餐直接能用」；其余厂商预设照常提供。onboarding 第 3 步同样如此呈现（官方推荐位 + 完整第三方列表）。~~在服务上线前，onboarding 默认推荐：中文界面 = 硅基流动，英文界面 = Groq（均有免费额度）。~~（2026-07-05 撤销：不对第三方作推荐标注，见 ADR-21。）
- **边界（开源信誉底线）**：客户端不为官方服务写任何专属私有协议逻辑（官方服务必须实现 OpenAI 兼容格式，走 `openai_compat` 通道）；不遥测、不上报使用量到官方服务器（计量在服务端完成）；「零上报、可审计、随时可换供应商」的承诺不因商业化而打折——这正是与 Typeless 形成差异的信任根基。
- **后续细化**：官方服务的计费、合规、隐私承诺等细节在服务筹备期另行决策。

### ADR-18 · 品牌配色改版：纯墨 Monochrome（2026-07-05，项目所有者拍板，取代 ADR-10）

- **决策**：全面弃用紫罗兰/电光青/Aurora 渐变，改为**纯墨（Ink）配色**——黑白灰三色构成整个界面（呼应域名 typex.ink），唯一保留的彩色是录音红/错误红（隐私功能色，不可灰化）。层级靠明度与结构（实底/灰底/发丝线/下划线/字重）表达；选中态用灰底 `--sel-bg` + 字重（禁止反色实底与粗黑描边，刺眼）；主按钮是页面上唯一的实底反色锚点。
- **图标与 logotype**：只用纯黑/纯白、无灰无渐变；浅色环境黑底白 glyph、深色环境白底黑 glyph；`Typex` 五个字母永远同色（浅色全黑、深色全白）。
- **评估过程**：候选含紫罗兰（原案）、靛蓝、正蓝、墨蓝 `#33415C`、松墨绿、赭墨、纯墨；经界面原型逐一对比后选定纯墨。完整 token 与对比度校验见 [04 设计系统](04-design-system.md)。

### ADR-19 · 增加主页窗口：统计 + 历史（2026-07-05，项目所有者拍板）

- **决策**：参照 Typeless 的主页形态，新增按需打开的**主页窗口**（880×560，左侧边栏导航：首页 / 历史记录）：首页页签 = 标题 + 快捷键提示 + 统计卡组（总口述时长 / 口述字数 / 节省时间 / 平均语速）+ 最近记录；历史记录页签合并原独立历史窗口。统计由本地历史库聚合（历史表增记每条会话的录音时长与字数），零上报；关闭历史即停止累计并隐藏统计。不含任何商业化模块（升级/推荐位——软件免费开源，[ADR-16](08-decisions.md)）。
- **边界**：「安静的工具」原则不变——无常驻主窗口，主页从托盘「主页…」/Dock 按需打开。界面规格见 [05 §8](05-ux-spec.md)。

### ADR-20 · 本地模型兜底：零配置模式（2026-07-05，项目所有者拍板，扩展 ADR-13）

- **背景**：BYOK 要求用户先配置第三方 API 端点（STT + LLM）才能使用，是最大的上手门槛。2026-07 调研确认端侧小模型已能覆盖大部分日常需求：SenseVoice-Small（int8 约 230 MB，非自回归，CPU 实时数倍速，中/英/日/韩/粤）质量接近云端；Qwen3-1.7B 级小模型（Q4 约 1.2 GB）足以胜任 F-9 整理这类窄任务。
- **决策**：新增**本地模型 Provider**，作为未配置 API 时的默认兜底（「零配置模式」）：
  - **STT**：sherpa-onnx（官方 Rust crate，静态链接）+ SenseVoice 为主引擎；whisper.cpp（whisper-rs）为多语言补充引擎。
  - **LLM**：llama.cpp（llama-cpp-2 绑定）+ Qwen3-1.7B 级 GGUF（Q4_K_M），零配置路径只覆盖整理与翻译两个槽位；问答槽不做本地兜底，未配置问答模型时助手功能显示引导配置，但用户可在设置中手动选择本地 LLM。
  - **模型不随安装包分发**，应用内按需下载（HuggingFace + 国内 ModelScope 镜像双源，校验和 + 断点续传）。
  - **产品呈现**：本地 Provider 是模型服务列表中的普通预设（「本地 · 离线」），与云端预设同列可插拔、槽位级自由混搭，不改变 BYOK 架构；onboarding 第 3 步提供「使用本地模型」一键选项。
- **节奏**：Provider 抽象保持兼容；优先落地本地 STT（收益最大、质量无折扣），本地 LLM 整理/翻译随后。本决策取代 ADR-13 的范围定义（从「本地转写」扩为「本地模型兜底」），路线与双引擎选型继承自 ADR-13。
- **边界**：本地推理零网络请求（下载模型除外）；延迟预算——整理短句 ≤ 500 ms 可达，模型常驻内存或录音开始时预热；安装包体积增量（推理引擎静态链接）约 30–60 MB。

### ADR-21 · 厂商中立与多配置档案快速切换（2026-07-05，项目所有者拍板）

- **厂商中立**：撤销「onboarding 默认推荐：中文界面 = 硅基流动、英文界面 = Groq」的口径（原 ADR-16 附带呈现）。Typex 是自由开源软件，未与任何第三方厂商有商业关系，界面上对第三方作「推荐」标注容易被误解为广告/回扣。onboarding 第 3 步的云端路径 = 直接填 API 端点 + 密钥；预设下拉仅作表单填充器，全部厂商平等罗列、无推荐徽标。唯一例外是未来的 Typex 官方服务（「官方 · 开箱即用」，ADR-16 边界不变）与「本地 · 离线」（ADR-20，属产品能力而非第三方）。
- **服务配置池 + 快速切换**：模型服务页维护全局 `profiles[]` 配置池，功能槽位只保存 `active_profile` 指针；同一个 LLM profile 可同时分配给文本整理、翻译和问答，新增不覆盖旧配置。典型场景：翻译与问答共用同一个远端 LLM，或日常本地小模型、重任务切云端大模型。切换入口：① 设置-模型服务页功能分配 ProviderCard「切换 ▾」下拉（✓ 标当前 + 新建配置）；② 托盘菜单「模型 ▸」子菜单，按 听写/文本整理/翻译/问答 四组列出兼容服务点选即切，无需打开设置。切换即时生效，作用于下一次会话。

### ADR-22 · 本地模型选型：Qwen 全家桶 + 硬件分档推荐（2026-07-05，项目所有者拍板，细化 ADR-20）

- **背景**：ADR-20 拍板本地兜底时选型为 SenseVoice + Qwen3-1.7B。随后确认 Qwen 已有完整端侧全家桶（均 Apache 2.0）：**Qwen3-ASR**（2026-01，0.6B/1.7B，52 语言 + 22 中文方言，1.7B 为开源 ASR SOTA，官方 ggml-org GGUF 可跑 llama.cpp；官方清单当前提供 Q8_0/bf16，Typex 选 Q8_0）与 **Qwen3.5 小模型系列**（2026-03，0.8B/2B/4B/9B，GGUF 齐全，4B≈上代 80B 水平）。用户反馈内置可下载模型偏小后，补充高配手动模型：Whisper large-v3 int8（sherpa-onnx 导出，约 1.8 GB）以及 Qwen3 14B / 30B-A3B / 32B Q4_K_M（llama.cpp GGUF）。
- **决策一（选型）**：本地模型默认采用 **Qwen 家族**——STT = Qwen3-ASR（0.6B/1.7B GGUF，Q8_0），LLM = Qwen3.5（0.8B/2B/4B/9B GGUF，Q4_K_M）。**SenseVoice-Small（sherpa-onnx）保留为轻量档 STT**：非自回归、CPU 实时数倍速，是弱机器上唯一能保证实时的选项（Qwen3-ASR-1.7B 纯 CPU 低于实时 RTFx≈0.49，0.6B 仅勉强实时）。**Whisper large-v3 int8** 作为高配精度 STT，质量优先但 CPU 低于实时，不参与零配置推荐档。内置 LLM 清单额外加入 SmolLM3-3B、Granite 3.3 2B Instruct、Phi-4 Mini Instruct，以及 Qwen3 14B / 30B-A3B / 32B，覆盖轻量英文/代码/通用指令与高配问答候选；这些模型不改变首次下载分档，只作为设置页可选模型。
- **决策二（硬件分档）**：下载时**按设备性能自动推荐档位**（检测 RAM / CPU 核数 / GPU 加速可用性：Metal/CUDA/Vulkan），用户可手动改档：
  - **轻量**（RAM < 8 GB 或无 GPU 且弱 CPU）：SenseVoice-Small int8（230 MB）+ Qwen3.5-0.8B（约 0.6 GB），合计约 0.8 GB。
  - **标准**（8–16 GB RAM）：Qwen3-ASR-0.6B Q8_0（主模型 + mmproj 约 1.0 GB）+ Qwen3.5-2B Q4_K_M（约 1.3 GB），合计约 2.3 GB。
  - **性能**（≥ 16 GB 且有 GPU 加速）：Qwen3-ASR-1.7B Q8_0（主模型 + mmproj 约 2.5 GB）+ Qwen3.5-4B Q4_K_M（约 2.7 GB），合计约 5.3 GB。
  - 档位只是首次下载的默认组合；下载后各槽位仍可在模型库中自由更换单个模型。
  - **高配手动模型**（不自动下载）：Whisper large-v3 int8（STT，约 1.8 GB，RAM ≥ 16 GB）；Qwen3-14B Q4（约 9.0 GB，RAM ≥ 32 GB）；Qwen3-30B-A3B Q4（约 18.6 GB，RAM ≥ 32 GB）；Qwen3-32B Q4（约 19.8 GB，RAM ≥ 48 GB）。
- **工程注意**：llama.cpp 音频(qwen3vl)支持仍标 experimental、有长音频 bug——用现有 VAD 切片规避（短分段转写本来就是 F-1 的路径）；1.7B STT 仅在 GPU 加速可用时提供。Whisper large-v3 当前走 sherpa-onnx CPU provider，界面不承诺 GPU 加速；后续如启用 CUDA/CoreML 需单独做运行时能力探测。问答槽不做本地兜底的口径不变（ADR-20）；若用户设备属性能档，设置中允许**手动**把问答槽指向本地大档模型（默认仍为空 + 配置引导）。
- **影响**：whisper.cpp 从「补充引擎」降为可选扩展（Qwen3-ASR 的 52 语言覆盖已够长尾，Whisper large-v3 已由 sherpa-onnx 覆盖）；Whisper large-v3-turbo、NVIDIA Parakeet、Moonshine 等模型需新增运行时或稳定导出后再进入内置下载清单。模型库清单（[03 §8](03-model-providers.md)）按档位组织，并支持用户托管导入清单外模型。

### ADR-23 · 助手交互重构：回答弹窗只在有回答要展示时出现（2026-07-06，项目所有者拍板）

- **背景**：原设计中助手键任何用法都会呼出「Spotlight 式助手面板」（输入条 + 动作行 + 📌 固定），改写型结果也先进面板再替换。实际使用中改写场景（选中文本 + 口述改写指令）根本不需要面板——弹窗成了多余的中间步骤；而面板的打字输入、动作按钮又把它推向「迷你聊天工具」，与「安静的工具」定位冲突。
- **决策**：助手面板降级为**只读回答弹窗**，仅在两种情况出现：① 对选中文本**提问**（`ANSWER:` 信号，流首部判定）；② **无选区**唤醒助手提问。改写型指令全程不弹窗，结果直接替换选区，反馈走 HUD 完整流水线（与听写/翻译一致）。
- **弹窗形态**：指令回显 + 流式 Markdown 回答 + ✕ 关闭；无输入条、无麦克风键、无动作按钮、无 📌；位置 = 选区下方（无选区时屏幕上 1/3 居中）；失焦（焦点切到其他应用）自动关闭。
- **连带简化**：助手键短按 = 切换式录音（与听写键语义一致，「仅呼出面板」废除）；托盘「打开助手」菜单项移除；打字提问入口（`ask_assistant` command）移除；设置「结果处置（自动替换/预览确认）」移除（改写恒直接替换）。
- **影响**：02 F-3、05 §2/§4/§7、03 §3.4、07 窗口表与 §10.1、08 场景表同步更新。

### ADR-24 · Windows 首发边界与分发链（2026-07-10，项目所有者拍板）

- **支持范围**：Windows 10 22H2+ 与 Windows 11，首发只构建 `x86_64-pc-windows-msvc`。不在首发维护 ARM64、32 位、MSI、MSIX、Microsoft Store 或 winget。
- **系统能力**：全局键采用原生 `WH_KEYBOARD_LL`，文本注入采用 SendInput，选区主路径采用 UI Automation。HUD 使用 Win32 no-activate/tool-window/topmost 语义，并按前台窗口所在 monitor 的 work area 与 DPI 定位。
- **安全与降级**：Typex 不自动提权。目标窗口因 UIPI 处于更高完整性级别、无焦点或不支持选区接口时，保住结果并降级到剪贴板/普通提问，给出明确提示；不得静默失败或注入到错误窗口。
- **音频与推理**：WASAPI 输入覆盖设备实际报告的 sample format；本地模型保证 CPU 路径，Windows GPU 统一走 Vulkan，不维护 CUDA 构建矩阵。缓存记录实际 load mode，并用独占 inference lease 串行同一模型的推理；仅 GPU-loaded 模型的运行初始化/decode 失败可自动 CPU 重试一次。CPU 重载必须清空模型设备并关闭 context/mtmd 的全部 GPU offload，先释放失败 GPU 代际，再在缓存锁外加载。LLM 只在首个可见 delta 前从头重试，已输出后不得重放；ASR 非流式可整次重试。输入/模型契约错误、无 GPU 或 CPU 路径错误不盲目重试。
- **安装器**：只产出 NSIS x64 用户安装器，WebView2 使用 Evergreen Bootstrapper；卸载默认保留设置、历史与模型，由应用内入口负责主动清理用户数据。默认 feature 的 sherpa/ONNX 四个 DLL 采用 app-local 分发；VC++ 2015-2022 x64 runtime 从构建机 Microsoft VC tools 的合法 redistributable 目录复制，优先当前 `VCToolsRedistDir` 或与 `VCToolsInstallDir` 同版本的目录，未显式提供当前 toolchain 时按 DLL `FileVersion` 选择最新完整集合，CRT 与 OpenMP 不得跨版本混装；Vulkan loader 从固定版本且校验 SHA256 的 Apache-2.0 包提取并随附许可。所有运行库与 EXE 同目录，使无系统级 VC runtime/Vulkan loader 的干净机器仍可启动，并在没有 Vulkan ICD/GPU 时进入 CPU fallback。
- **更新与密钥**：Windows 与 macOS 复用仓库现有 Tauri updater 密钥。Windows 采用 Tauri 2 原生 updater 产物，直接对作为安装器的 NSIS `.exe` 生成 `.exe.sig`，不保留 Tauri v1 legacy zip；各平台资产在上传前必须用配置公钥实际验签。各平台先上传唯一 manifest fragment，再聚合为单一 stable/nightly `latest.json`；平台键、资产名和通道不得互相覆盖。
- **发布者签名**：Tauri updater `.sig` 负责更新完整性，Windows Authenticode 负责发布者身份，二者不能互相替代。SignPath Foundation 审批时间属于外部依赖，因此功能适配与 CI artifact 允许暂时 unsigned，但必须保留内层 EXE 与外层 NSIS 的稳定签名插入点；正式大范围公开分发前重新评估。自签名证书不作为公共信任替代方案。
- **验收**：受控 Win32/UIA harness 是系统契约门槛；记事本、Edge、VS Code、Windows Terminal 是代表性应用矩阵。微信、Slack、Word、Chrome 等只作非阻塞兼容抽测，除非问题导致崩溃、数据/剪贴板损坏、隐私泄露、错窗口注入，或能在 harness 复现为 Windows 契约错误。

### ADR-25 · VAD 双路径与 Windows 即时起音候选录音（2026-07-12）

- **背景**：固定能量门限会漏掉轻声或噪声环境中的语音；Windows 为保护 `Ctrl+C`/AltGr 等普通组合键而延迟 75 ms 提交 `TriggerDown`，若确认后才打开麦克风，会形成固定起音缺口。
- **决策**：schema v7 增加神经网络/能量双 VAD，升级后默认使用带完整内嵌权重的 `silero-vad-crs 0.4`，失败时降级到用户能量门限；首尾保护采用 300/150 ms，连续弱信号提供 90 ms 保底。Windows 在原始触发键按下时启动带 token 的内存候选流，确认后原位提升，普通组合键、暂停、配置更新、hook 终态与退出均匹配取消。
- **隐私与边界**：候选阶段不可见、不落盘、不进历史、不发 Provider/电平事件，取消即释放；不采用常驻麦克风，因此设备真正开始回调前的硬件启动时间仍不可追溯。录音开始时快照 VAD 配置，失败重试与长录音切片保持同一判断口径。
