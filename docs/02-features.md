## F-1 语音输入（听写）

### 描述

用户在**任意应用**中，按下听写快捷键说话，松开后语音被转成文字、经**轻量文本整理模型去噪**（去语气词、修标点、保留改口后的最终意图），再注入到当前光标位置。这是 Typex 的主干路径，其他功能都是这条流水线的变体。

> 为什么必须有「整理」这一级：用户口语天然杂乱（语气词、重复、中途改口），原样上屏并不可用。「STT 原始转写 → 轻量模型整理」两级流水线是本品类的核心价值，规格详见 F-9。

### 流水线

```
按键按下 ──▶ 开始录音（HUD 出现，波形反馈）
按键松开 ──▶ 停止录音 ──▶ [所选 VAD 裁剪静音] ──▶ STT Provider 转写
        ──▶ 文本整理（F-9，默认开启，可一键切「原样模式」）
        ──▶ 文本注入（光标处）──▶ HUD 成功反馈后隐藏
```

### 规格

- **触发方式**（详见 F-5）：
  - **长按说话（push-to-talk）**：按住 ≥ 350 ms 进入长按模式，松开即结束。这是主推交互。
  - **短按切换（toggle）**：按一下开始，再按一下结束。两种模式同一个键自动区分，无需配置。
  - `Esc` 或点击 HUD 取消按钮 = 放弃当前尚未提交的流程，不产生任何输出；覆盖录音、录音收尾、转写、处理、注入提交前与失败态。全局 `Esc` 仅在成功认领当前会话取消权时吞掉该次物理按键的 down、自动重复与配对 up；空闲、设置关闭或注入已经提交时完整透传。设置中可关闭全局 `Esc` 取消，但不影响回答弹窗自身的关闭行为。
- **录音**：16-bit PCM，按设备原生采样率采集后重采样为 16 kHz 单声道（见 [06 代码架构 §7.4](06-code-architecture.md)）。录音期间 HUD 显示实时波形（麦克风是否在工作必须一眼可见）。
- **松键反馈**：录音结束后先发布 `Transcribing` 状态，再在阻塞工作线程中关闭音频流、重采样、执行 VAD 与 WAV 编码；快捷键结束到 HUD 切换为「正在转写」目标 ≤ 100 ms，音频收尾期间仍可响应取消，且取消后不得启动 STT。
- **语音活动检测（VAD）**：设置中可选「神经网络」（默认）或「能量门限」。神经网络路径使用内嵌权重的 Silero VAD，每 512 个 16 kHz 样本推理；进入门限默认 `0.50`，退出门限为 `max(进入门限 - 0.15, 0.01)`，最短语音 90 ms、结束静音 100 ms。能量路径以 30 ms RMS 帧判断，默认门限 `0.010`。两种模式都从首个语音区间向前保留 300 ms、从最后语音区间向后保留 150 ms；神经网络初始化或推理失败时自动降级到用户设置的能量门限。
- **弱声保底**：VAD 未检出语音但录音中存在连续至少 90 ms 的弱信号时，提交未裁剪音频给 STT，避免轻声起音被误删；真正静音仍返回 `NoSpeech`。录音开始时快照整组 VAD 配置，失败重试与长录音切片沿用同一快照。
- **无时长硬上限**：这是对 Typeless「6 分钟上限」的直接回应。实现上：
  - 录音超过 Provider 单次上限（如 OpenAI 25 MB）前自动切片，切片边界选在 VAD 静音处，分段转写后拼接，用户无感。
  - HUD 显示已录制时长；超过 10 分钟给出温和提示（非中断）。
- **语言**：默认「自动检测」；可在设置中固定语言（传给 STT 的 `language` 参数，能提升准确率与速度）。
- **注入**：默认「剪贴板粘贴 + 恢复剪贴板」策略；按平台自动选择注入后端，允许高级用户切换（见 [06 代码架构 §7.5](06-code-architecture.md)）。焦点应用不可写（无输入焦点）时，结果自动进入剪贴板并由 HUD 提示「已复制，Ctrl+V 粘贴」。
- **失败处理**（铁律：不丢话）：
  - 麦克风固定设备缺失、启动失败或录音中被拔出：录音态立即结束并进入可见失败态，释放当前音频流；修复设备后可直接重试录音，不等待用户再次按下或松开快捷键。
  - 网络/服务错误：指数退避自动重试 2 次；仍失败则 HUD 显示错误与「重试」按钮，**本次录音音频保留在内存/临时目录**直至用户放弃。
  - 转写结果为空（纯静音）：HUD 提示「没有听到声音」，附麦克风选择快捷入口。

### 验收标准

1. 受控文本输入 harness 必须正确接收中英文、标点、emoji 与换行；平台代表应用至少覆盖 macOS 原生文本控件，及 Windows 的记事本、Edge、VS Code、Windows Terminal。微信 / Slack / Word 等只作兼容抽测，不是支持声明的前置条件。
2. 5 秒中文短句：松开按键到文字上屏 P50 < 2 s（含整理环节；「原样模式」下 < 1.5 s。以 Groq/豆包极速版级别的 Provider 测量）。
3. 含语气词、重复与中途改口的口语样句，上屏结果为整理后的干净文本（测试样例集见 F-9）。
4. 断网时给出明确错误且音频不丢；恢复网络后点重试成功。
5. 15 分钟连续口述不中断、不丢内容。
6. 非美式键盘布局（如德语 QWERTZ、中文输入法激活状态）下注入内容无乱码。
7. Windows 目标窗口无焦点或因 UIPI 处于更高完整性级别时，不尝试自动提权；结果完整复制到剪贴板并显示可操作提示。
8. Windows 默认快捷键按下后立即发音时，75 ms 语义确认窗口不得截掉起音；轻声起音可通过弱声保底送达 STT，纯静音仍不得发起 Provider 请求。

## F-2 语音翻译

### 描述

用户按住**翻译快捷键**说 A 语言，松开后 Typex 注入 B 语言译文。典型场景：说中文 → 上屏英文。

### 流水线

```
翻译键按下 ──▶ 录音（HUD 显示翻译模式徽标 + 目标语言）
松开 ──▶ STT 转写（源语言自动检测）──▶ 文本整理（若 F-9 开启）──▶ 翻译 Provider（LLM）──▶ 注入译文
```

### 规格

- **语言配置**：设置中配置**源语言**（你说话使用的语言，默认跟随界面语言）与**目标语言**（要翻译成的语言，默认 `英文`）。HUD 上显示当前方向（如 `中 → EN`），点击可快速切换最近使用的语言。
- **双向翻译（默认开启）**：检测到说的是目标语言时自动译回源语言（说中文→出英文，说英文→出中文）——覆盖「双语工作者」主场景，免去手动切方向。
- **文本整理复用**：翻译模式共享 F-9「文本整理」开关和系统提示词；默认开启，STT 后先整理再翻译。关闭文本整理时，原始转写直接进入翻译请求。整理失败时直通原始转写，不阻断翻译。
- **翻译 Prompt**：内置 system prompt 只定义译者角色、忠实性和输出约束，高级设置允许用户完整覆盖；本次翻译的语言方向、双向开关、目标应用和文本由程序生成固定 XML user message，不能在设置中修改。模型只输出译文本身，不解释、不加引号，并保留原文语气与格式。
- **目标应用上下文**：录音开始时采样的目标应用名写入整理与翻译请求的 `<target_app>` 可选元素，用于语气和术语判断；平台不支持或读取失败时省略该元素。
- **翻译模型**：使用「翻译模型」配置槽（F-4），支持 OpenAI 兼容 / OpenAI Responses 两种格式。翻译走一次普通的 LLM 调用（非 Agent），要求低延迟——推荐用户配置轻量模型。
- **流式**：翻译结果采用流式接收，但**注入是一次性的**（收完再注入，避免在目标应用中逐字打断用户）。
- 失败处理同 F-1；额外地：若 STT 成功但翻译失败，HUD 提供「注入原文」的降级选项——转写结果不浪费。

### 验收标准

1. 说中文出英文、说英文出中文（双向模式）各 20 句样本，无一句注入原文未翻译。
2. 松开按键到译文上屏 P50 < 2.5 s（转写 + 翻译两跳）。
3. 口述含列表结构（「第一…第二…」）时译文保留结构。

## F-3 文本处理与语音问答

> 范围说明：Typex **刻意不做完整的 AI Agent**（无工具循环、无本地文件/命令操作、无多轮会话）。市面上已有 Codex、Claude Code 等成熟 Agent 产品，重造轮子意义不大。F-3 的两个运行时请求都是**单次 LLM 调用**：a 是「指令 + 选中文本 → 改写或回答」，b 是「无选区问题 → 回答」。此决策记录于 [08 决策记录 ADR-1](08-decisions.md)。
>
> 交互原则（[ADR-23](08-decisions.md)）：**回答弹窗只在「有回答要展示」时出现**——改写型指令直接替换选区、不弹窗；只有「对选区提问」和「无选区提问」两种情况弹出只读回答弹窗。

### F-3a 选中文本 + 语音指令（原地处理）

**描述**：用户在任意应用中选中一段文本，按住**助手快捷键**口述需求（「翻译成日语」「改得正式一点」「精简一半」「把这段 JSON 格式化」），松开后 Typex 将处理结果**原地替换**选中文本——**全程不弹窗**，反馈走 HUD（与听写/翻译一致的完整流水线）。

**流水线**：

```
选中文本 ──▶ 助手键按下 ──▶ 录音
松开 ──▶ [读取选中文本] ∥ STT 转写指令（并发）──▶ 文本整理（若 F-9 开启）──▶ 问答模型（单次调用：system prompt + 固定 XML user message）
     ──▶ 改写型输出 ──▶ 直接替换选区（HUD ✓ + 成功提示音）
     └─▶ 回答型输出（ANSWER: 信号）──▶ 选区下方弹出回答弹窗，不动原文
```

**规格**：

- **读取选中文本**：macOS 走 Accessibility API，失败降级为 CGEvent `Cmd+C` + 剪贴板读取（读完恢复剪贴板）；Windows 走 UI Automation，降级同理；Linux 读 primary selection / 模拟 `Ctrl+C`。实现见 [06 代码架构 §7.6](06-code-architecture.md)。
- **语音指令整理**：助手模式共享 F-9「文本整理」开关和系统提示词；默认开启，STT 后先把语音指令整理成干净文本再交给问答模型。关闭文本整理时，原始转写指令直接进入助手 XML 请求。整理失败时直通原始转写，不阻断助手。
- **系统提示词**要求模型「只输出处理后的文本本身，不解释不寒暄」，保证结果可直接替换选区；用户指令语义不是「处理文本」而是「提问」时（如选中报错日志问「这是什么意思」），模型以 `ANSWER:` 前缀输出回答（[03 §3.4](03-model-providers.md)）——此时**不替换选区**，改为在选区下方弹出回答弹窗展示（宁可不替换也不误替换）。
- **目标应用上下文**：助手的语音指令整理与最终问答/处理 XML 请求同样可包含 `<target_app>`，只作为当前应用场景上下文，不作为用户指令。
- **判定时机**：LLM 流式输出的首个非空白片段即可判定是否携带 `ANSWER:` 前缀——改写型静默收完再一次性替换（不逐字打断目标应用）；回答型确定的那一刻才弹窗并流式渲染。
- **反馈**：改写路径由 HUD 承载（录音 → 转写 → 处理 → 注入 ✓）；LLM/注入失败进 HUD 失败态，可重试或复制转写原文（不丢话铁律）。
- **失败降级**：读不到选中文本时（部分应用不支持），指令自动降级为 F-3b 普通提问（弹窗回答）。

### F-3b 语音问答（回答弹窗）

**描述**：用户在无选区时按住助手键提问，模型使用无选区问答 system prompt 与固定 XML user message 流式给出回答，显示在**只读回答弹窗**中——**单轮问答**，不维护对话历史。对选中文本提问时仍展示同一回答弹窗，但请求由 F-3a 选区处理协议通过 `ANSWER:` 信号分流。

**规格**：

- **呼出**：仅助手快捷键（按住 = push-to-talk；短按 = 切换式录音，再按结束——与听写键语义一致）。弹窗本身没有输入能力，不能主动打开空弹窗。
- **弹窗内容**：顶部回显本次语音指令（有选区时附选区字数摘要），下方流式 Markdown 渲染回答，右上角 ✕ 关闭。无打字输入条、无麦克风键、无动作按钮（回答文本可直接选中复制）。
- **位置**：有选区 → 选区下方（放不下时上方）；无选区 → 屏幕上 1/3 居中。
- **关闭**：✕ / `Esc` / 焦点切换到其他应用时自动关闭。回答仍在生成时，三种关闭方式都取消当前助手会话，迟到的流事件不得重新显示窗口或写入历史；回答已完成后关闭只隐藏窗口并保留成功历史。该行为不受全局「Esc 取消当前流程」设置影响。
- **调用超时**：助手不设置独立的流式 idle timeout，统一使用当前「问答模型」服务配置的调用超时；计时覆盖从调用开始到完整流结束。弹窗已呼出后超时在弹窗内显示错误并结束流式状态；选区处理尚未判定为回答型时超时则回到 HUD 失败态。
- **单轮语义**：每次提问独立（不携带上一问）；新提问的弹窗覆盖上一条回答。多轮对话明确为非目标（若未来需要，交给专业工具）。
- **模型**：使用「问答模型」配置槽（F-4），支持 OpenAI 兼容 Chat Completions / OpenAI Responses 两种格式，流式输出。
- **历史**：仅完整成功的回答写入一条助手记录（语音问题/指令 + 完整回答 + 录音时长 + 目标应用）；流式错误、空回答、用户取消、部分流、选中文本和音频均不落库。改写型指令仍只在成功注入后记录。

### 验收标准

1. 在受控 AX/UIA harness 与平台代表应用中选中文本 → 口述「翻译成英文」→ 原地替换成功，**全程无弹窗**，HUD 显示完整流水线并以 ✓ 结束；不暴露文本选区 API 的应用必须可靠降级为剪贴板读取或普通提问。
2. 选中报错日志口述「这是什么原因」→ 选区下方弹窗展示回答且**不**替换选区。
3. 无选区按住助手键提问 → 屏幕上 1/3 居中弹窗展示回答。
4. 弹窗出现后点击其他应用（焦点切走）→ 弹窗自动关闭。
5. 读取选区失败的应用（如部分终端）自动降级为普通提问，流程不中断。
6. 弹窗流式输出首 token < 1.5 s（取决于模型，测试用主流云端模型）。
7. 连续两次提问互不影响（无上下文泄漏）。

## F-4 模型服务配置（BYOK）

### 描述

用户在设置中维护一个**模型服务配置池**，四个功能槽位只保存「当前选择哪个服务配置」的指针。同一个远端 LLM 服务配置可以同时分配给文本整理、翻译和问答，无需重复填写 Base URL、模型名和密钥。

| 功能槽位 | 用途 | 可选择的服务能力 | 接入格式 |
|---|---|---|---|
| **语音转文字** | F-1/F-2/F-3 的语音识别 | STT 服务配置 | ① OpenAI 兼容转录格式（multipart `/audio/transcriptions`）② 火山引擎/豆包（极速版 flash 接口，AppKey + AccessToken 双凭据） |
| **文本整理** | F-9 去噪整理（F-1 主干） | LLM 服务配置（推荐轻量快模型） | ① OpenAI 兼容 Chat Completions ② OpenAI Responses |
| **翻译模型** | F-2 翻译 | LLM 服务配置（轻量~中档） | 同上 |
| **问答模型** | F-3 文本处理与问答 | LLM 服务配置（可选更强模型） | 同上 |

> 降低配置负担：onboarding 只要求配 STT + 一个 LLM 端点，后三个 LLM 功能默认选择同一个 LLM 服务配置；用户可随时把任一功能切到池中另一个服务。「一个密钥就能跑起来」是底线体验。

### 规格

- **预设模板**：内置常见服务的一键预设（OpenAI、Groq、SiliconFlow、火山引擎/豆包、DeepSeek、OpenRouter、Ollama/自建），选预设后只需填密钥；同时永远允许「自定义」纯手填。预设仅是表单填充器，不是锁定项；**onboarding 与设置中不对任何第三方厂商作「推荐」标注**（保持中立，[ADR-21](08-decisions.md)）。**「本地 · 离线」预设**（F-12）：无需密钥，选中后引导下载模型；STT/整理/翻译可被零配置路径自动指向本地，问答槽不自动兜底但设置中可手动选择本地 LLM。**Typex 官方 STT 服务**上线后作为预设列表首位（标注「官方 · 开箱即用」，走标准 `openai_compat` 通道、无任何客户端特权，[ADR-16](08-decisions.md)）。
- **连接测试**：每个服务配置有「测试」按钮——STT 服务发送一段内置的 2 秒测试音频，LLM 服务发送一条 ping 消息；显示往返延迟与结果。
- **密钥存储**：密钥与其他模型服务配置项一起保存在本机配置文件中；诊断导出与日志必须脱敏。见 [06 代码架构 §9](06-code-architecture.md)。
- **配置池 + 功能分配（[ADR-21](08-decisions.md)）**：服务配置池保存多个 STT / LLM 配置；功能分配区只选择池中兼容的服务。新增配置不覆盖旧配置——典型场景：翻译和问答共用同一个远端 LLM，或日常本地小模型、重任务一键切云端大模型。切换入口：① 设置-模型服务页功能分配卡片的「切换 ▾」；② **托盘菜单「模型 ▸」子菜单**（按 听写/文本整理/翻译/问答 分组列出兼容服务，点选即切，无需打开设置，见 F-6）。切换即时生效，作用于下一次会话。
- 每个 STT / LLM 服务配置可独立设置一个**全局调用超时**（默认 60 秒）：STT 从转写调用开始计时，直到完整文本返回；LLM 从调用开始计时，直到完整流式响应结束。同一档案被多个功能复用时统一生效，不得在功能层再叠加额外时限。每个服务配置还可设置自定义 HTTP Header（应对网关/代理场景），STT 服务可设置语言提示与温度。

### 验收标准

1. 四个功能槽位分别选择不同服务（如 STT=豆包、整理=DeepSeek、翻译=Gemini、问答=OpenAI）时全功能正常；仅配 STT + 单一 LLM 时三个 LLM 功能正确共用同一个服务配置。
2. 密钥在 `settings.json` / 日志 / 崩溃报告中均不可见。
3. 错误的密钥/端点在「测试」时给出可读的错误说明（401/404/超时 分别有针对性文案）。

## F-5 快捷键系统

- 三个全局快捷键：**听写**、**助手**、**翻译**（默认为全修饰键三角方案：右⌘/右Ctrl、右⌥/右Alt、两键同按；详见 [05 UX 规格 §7](05-ux-spec.md)）。
- 三项配置中的每个按键数组分别表示一个可编辑的**完整 chord**，不是多个候选键：只有数组内全部物理键同时按住才触发。翻译 chord 可独立于听写与助手直接触发；默认值仍是听写键与助手键的有序组合。
- 三组 chord 必须非空。听写与助手不能相同或互为子集；翻译只不能与听写或助手完全相同，允许包含前两项的按键，也允许成为其中一项的严格子集。若当前 chord 是随后补全 chord 的严格子集，较长 chord 接管当前模式并保留音频；没有包含关系的其他已完成 chord 不改变本次手势最先启动的模式。设置界面必须阻止保存非法组合，后端也必须拒绝非法更新；读取到非法配置时仅把热键恢复为当前平台默认值，其他设置保持不变。
- 每个键支持「长按 push-to-talk / 短按 toggle」双模（同键自动区分，阈值 350 ms，可调）。
- 支持**单个修饰键**作为触发键（如仅「右 ⌘」「右 Alt」）——这是品类标准体验，需要底层键盘监听而非普通热键 API（实现见 [06 代码架构 §7.3](06-code-architecture.md)）；**组合键让路规则**保证触发键的日常组合用法（⌘C 等）零干扰。
- Windows 在单修饰键 75 ms 语义确认窗口开始时立即启动仅驻内存的候选录音；确认 Typex 手势后以同一 token 提升为正式录音，不重开设备。普通键、AltGr、暂停、配置更新、hook 失败或退出会静默取消匹配候选，不显示 HUD、不播放提示音、不写历史或磁盘，也不向 Provider/电平事件发送任何候选数据。
- 自定义普通键按物理位置记录，与当前键盘布局及输入法输出字符无关；历史版本保存的后端别名在读取时迁移到稳定 `KeyId`，界面标签仍按 macOS/Windows/Linux 惯例显示。
- 冲突检测：录制快捷键时检测与系统/常见软件的已知冲突并警告。
- Wayland 降级：无法全局监听时自动引导用户走 Portal 授权或 compositor 绑定方案（见 [06 代码架构 §8](06-code-architecture.md)）。

## F-6 托盘与常驻行为

- Typex 无主窗口，启动后驻留系统托盘/菜单栏。托盘图标反映状态：空闲 / 录音中 / 处理中 / 错误（图标规格见 [04 设计系统 §3.4](04-design-system.md)）。
- 托盘菜单：`复制上次结果`、`文本整理开关`、`翻译目标`、**`模型 ▸`（按 听写/文本整理/翻译/问答 分组列出兼容服务配置，点选即切，F-4 配置池的快速切换入口）**、`暂停 Typex`（临时禁用全局监听；进入暂停时静默取消当前录音，含短按 toggle，不中断已经进入转写/处理的会话）、`设置…`、`主页…`、`检查更新`、`退出`。
- 单实例：二次启动唤起设置窗口而非新进程。
- 开机自启：默认询问（onboarding 最后一步），可随时在设置中改；应用启动与设置变更只在系统注册状态和配置不一致时写入，已关闭且不存在注册项是正常状态，不得产生警告。Windows 注册项必须使用带引号的当前 EXE 完整路径；启用时自动修复仍指向旧 debug/安装目录的条目，关闭时删除残留条目，完全一致时不重复写注册表（[ADR-26](08-decisions.md)）。
- 自动更新：内置 updater（GitHub Releases 渠道），默认开启检查、手动确认安装（[ADR-11](08-decisions.md)）。首次生成设置时，开发版（SemVer prerelease，当前为 `-dev`）默认使用 nightly 通道，纯 `MAJOR.MINOR.PATCH` 正式版默认使用 stable 通道；用户手动选择通道后按保存值检查，不由构建类型覆盖。
- Windows 分发为 x64 NSIS 用户安装器，WebView2 使用 Evergreen Bootstrapper；保持免 UAC 的 per-user 安装。无历史安装且未用 `/D=` 指定目录时默认安装到 `%LOCALAPPDATA%\Programs\Typex`，覆盖升级与自定义目录继续原位安装，不自动搬迁。卸载默认保留设置、历史和已下载模型。Tauri updater 完整性签名与 Windows Authenticode 发布者签名相互独立（[ADR-26](08-decisions.md)）。

## F-7 历史记录与主页统计

- **默认开启**（[ADR-8](08-decisions.md)）：本地 SQLite 保存每次成功会话的转写文本、译文/处理结果/助手回答、时间、目标应用、**录音时长与字数**；**不保存音频或选中文本**（音频仅在失败重试期间临时保留）；数据永不离开本机。onboarding 与设置页均明示这一点，可一键关闭并清空。
- **主页**（主窗口，见 [05 §8](05-ux-spec.md)，[ADR-19](08-decisions.md)）：顶部统计卡片（总口述时长 / 口述字数 / 节省时间 / 平均语速，由历史库本地聚合）+ 历史列表（搜索、展开对照、复制、删除单条、一键清空）；设置保留期限（7/30/90 天/永久，默认 90 天）。助手问答记录可在列表中查看和复制回答，但全部排除在四项口述统计之外，避免回答长度污染输入效率口径。
- 关闭历史时统计停止累计并隐藏；**最近 1 条**结果仍可从托盘「复制上次结果」找回（内存级，重启即失）。

## F-8 首次启动引导（Onboarding）

五步向导（详细界面见 [05 UX 规格 §6](05-ux-spec.md)）：

1. **欢迎** — 品牌页（图标 + 口号，居中，无数据承诺文案——承诺见 README/官网）；左下角语言上拉框（简体中文/English，即时切换，兜底系统语言检测错误）。
2. **权限** — 按平台申请：麦克风、辅助功能（macOS）/ 输入监听；每项权限说明「为什么需要」，实时检测授权状态，未完成不可下一步（可跳过但功能受限并明示）。
3. **模型配置** — 两条路径：**「使用本地模型」一键选项**（下载 SenseVoice + 轻量 LLM 后 STT/整理/翻译功能指向本地服务配置，无需任何密钥，[ADR-20](08-decisions.md)）；或**连接云端服务**——两组表单直填：语音转文字（端点 + 密钥）、大语言模型（端点 + 密钥，三个 LLM 功能默认选择同一服务配置，可稍后在设置中分开），无预设入口、不作厂商推荐（[ADR-21](08-decisions.md)）。
4. **快捷键** — 显示听写、助手、翻译三组默认键位（全修饰键三角方案），三项均允许当场修改；内嵌一个练习输入框：「按住 右⌘/右Ctrl 说：你好，Typex」——首次成功上屏是 aha 时刻。
5. **完成** — 仅 ✓ + 「一切就绪」，保持干净；开机自启默认开启（设置-通用可关，Linux 包管理渠道除外）。点击完成时先持久化 `onboarding_done` 与自启选项，再创建/显示并聚焦主页；只有主页成功就绪后才关闭引导窗口，失败时保留完成页并允许重试。

## F-9 文本整理（去噪层）

**这是 F-1 语音输入的核心组成部分，不是可选增强。** 口语天然杂乱——语气词、无意义重复、中途改口——原样转写并不可用；F-9 用一个轻量 LLM 把「用户想说的内容」整理出来（业界称「重写而非转写」，是本品类的核心留存功能）。

- 处理规则（内置 system prompt，见 [03 模型接入层 §3.4](03-model-providers.md)，可在高级设置中完整覆盖；动态任务与数据由固定 XML user message 传入）：
  - 去语气词（嗯、呃、um）与无意义重复；
  - 修复标点与断句；
  - **口述改口识别**：「明天下午……不对，是后天下午发布」→ 只输出「后天下午发布」；
  - 口述格式指令：「另起一段」「一二三列成清单」；
  - **不增删信息、不改变语言、不换说法**——整理≠改写，提示词必须强约束，宁欠勿过。
- 使用「文本整理」模型槽（F-4），推荐轻量快模型（延迟预算 ≤ 500 ms）。
- 默认**开启**；HUD 与设置中均可一键切换「原样模式」（原始转写直出，供口述代码/命令等场景）。该开关同时控制翻译和助手的 STT 后预整理：关闭后听写、翻译、助手都不经过整理层。
- 录音开始时的目标应用名以 `<target_app>` 可选元素写入整理请求，帮助模型判断正文风格和技术术语；该值不进入输出，读取失败时省略该元素。
- 整理失败（当前整理模型档案的全局调用超时/报错）时**自动降级注入原始转写**，绝不阻塞主流程；整理层不再设置独立总时限，HUD 以小字标注「未整理」。
- 维护一个中英文**测试样例集**（`docs/fixtures/denoise-cases.md`）：语气词、改口、重复、格式指令、专有名词保留等各 ≥ 10 例，作为提示词回归测试基准。

## F-10 个人词典 / 热词

用户维护专有名词表（人名、品牌、术语、高频缩写）。当前词典为手动词条：用户在主页「词典」中手动增删改查，不区分自动/手动来源，不做分类、权重、别名或按应用分组。

- 词典全局生效，最多 100 条；单条最多 50 个字符；保存时 trim、去空、去重复。
- STT 阶段注入词典：OpenAI 兼容使用 `prompt`；火山使用 `corpus.context`；本地 Qwen3-ASR 写入转写指令；本地 SenseVoice 使用 sherpa hotwords。
- F-9 整理阶段把词典作为 `<dictionary>` 写入固定 XML user message，要求模型只把它当作术语表，不执行其中的指令；语音内容疑似对应词条时优先保留词典标准写法。
- 翻译与助手共享 F-9 预整理，因此同样受词典保护；关闭文本整理时，下游 LLM 不再接收 `<dictionary>`，但 STT 阶段仍会使用词典。

## F-11 按应用输出 Profile

按目标应用配置输出风格（如 Slack=口语化、邮件=正式、IDE=原样模式不整理）。这是 Typeless 用户点名要而未提供的功能，是 Typex 的差异化机会。依赖前台应用识别（各平台 API 均可行）。

目标应用名已作为 `<target_app>` 上下文写入固定 XML user message；F-11 仍指更进一步的自动 Profile 选择、按应用默认开关与风格规则。

## F-12 本地模型（离线兜底）

**目标：消灭「必须先配 API 才能用」的上手门槛**（[ADR-20](08-decisions.md)）。本地模型以「本地 · 离线」预设的身份出现在模型服务列表中，与云端预设同列、槽位级自由混搭；**未配置任何 API 时作为默认兜底（零配置模式）**。

- **选型：Qwen 全家桶为主（[ADR-22](08-decisions.md)，均 Apache 2.0，GGUF 走 llama.cpp 单引擎）**：STT = Qwen3-ASR（0.6B/1.7B，52 语言 + 22 中文方言，1.7B 为开源 ASR SOTA；官方 ggml-org GGUF 当前可用量化为 Q8_0/bf16，Typex 选 Q8_0）；整理/翻译 = Qwen3.5 小模型（0.8B/2B/4B，Q4_K_M）。SenseVoice-Small（sherpa-onnx，int8 约 230 MB，非自回归、CPU 实时数倍速）保留为轻量档 STT——弱机器上唯一能保证实时的选项。
- **硬件分档推荐**：下载时按设备性能（RAM / CPU / GPU 加速）自动推荐档位，可手动改档；探测结果只表达预期性能与推荐程度，不构成下载门槛。只要内置模型存在远程下载源，即使设备低于建议也始终允许下载；档位只影响首次下载组合，之后各槽位可在模型库自由换单个模型（[ADR-26](08-decisions.md)）：

| 档位 | 适用设备 | STT | 整理/翻译 LLM | 合计体积 |
|---|---|---|---|---|
| 轻量 | RAM < 8 GB 或弱 CPU 无 GPU | SenseVoice-Small int8（230 MB） | Qwen3.5-0.8B（约 0.6 GB） | 约 0.8 GB |
| 标准 | 8–16 GB RAM | Qwen3-ASR-0.6B Q8_0（主模型 + mmproj，约 1.0 GB） | Qwen3.5-2B Q4_K_M（约 1.3 GB） | 约 2.3 GB |
| 性能 | ≥ 16 GB + GPU 加速 | Qwen3-ASR-1.7B Q8_0（主模型 + mmproj，约 2.5 GB） | Qwen3.5-4B Q4_K_M（约 2.7 GB） | 约 5.3 GB |

- **问答槽不提供本地兜底**——未配置时助手指令走 HUD 失败态显示配置引导；但设置中允许用户手动选择本地 LLM 档案（性能档设备可选本地大档模型）。
- **模型管理**：不随安装包分发；应用内按需下载（HuggingFace + ModelScope 双源自动择优，校验和 + 断点续传），设置-模型服务页可删除已下载模型，并在模型管理页底部固定下载源（自动 / HuggingFace / ModelScope）。
- **运行时**：模型常驻内存或「录音开始时预热」可选；本地推理零网络请求。
- **Windows 加速**：CPU 路径始终可用；GPU 路径统一使用 Vulkan，不维护 CUDA 专用分发矩阵。缓存必须记录模型实际加载模式，同一模型实例的推理通过独占租约串行执行，避免并发请求在 fallback 时继续持有失败的 GPU 模型。仅当模型确实以 GPU offload 加载后，运行时上下文初始化或 decode 失败时自动改用严格 CPU 参数重载并从头重试一次：模型设置 `n_gpu_layers=0` 且设备列表为空，context 禁用 K/Q/V 与算子 offload，Qwen3-ASR 的 mtmd 同时禁用 GPU；释放失败 GPU 模型后才可在缓存锁外加载 CPU。本地 LLM 只允许在 ThinkingFilter 后尚未发出首个可见 delta 时重试，已发出 delta 后明确失败且不得重放；Qwen3-ASR 为非流式，可在整次转写返回前用 CPU 完整重试一次。GGUF/音频/提示词校验、上下文过长等输入错误、无 GPU、已经 CPU 加载或 CPU 重试失败均不得继续重试。
- **零配置路径**：onboarding 第 3 步提供「使用本地模型」一键选项（显示检测到的推荐档位与体积，下载后 STT/整理/翻译功能指向本地服务配置），见 F-8。
- Provider 抽象与模型下载管理器统一承载本地 STT 与本地 LLM；本地能力与云端能力可按功能槽位混搭。
